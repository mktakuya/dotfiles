- そのため、同音異義語の誤変換や音声認識由来の誤字が含まれる場合がある（例: 「CLAUDE」→「cloude」、「プロンプト」→「フロント」など）

### 表記ルール

コミットメッセージ、コードコメント、PR の説明文など、日本語の文章を書くときに適用する。

- 英単語／半角数字と日本語の間には半角スペースを入れる
- ただし以下は除く:
  - 文頭、文末、句読点、括弧と隣り合う箇所
  - 固有名詞以外の略語（例: EXEやDLL）
  - 連結して一塊となっている語（例: Webサイト、10px、1,000円）

## 思考・提案プロセス（BFS → DFS アプローチ）

設計・アーキテクチャに関わる議論や、複数のアプローチが考えられる課題に対して適用する。自明なタスク（typo 修正、単純なリファクタリング、明確に一択の実装など）ではこのプロセスを省略してよい。

ユーザーは「手堅く仕事を進める」ために、課題解決において**「幅優先探索（BFS）で合意してから、深さ優先探索（DFS）を行う」**ことを重視しています。コーディングエージェントは壁打ち相手として、以下のプロセスを強制・促進してください。

1. **幅優先探索 (BFS) フェーズ:**
   - すぐに単一の解決策の詳細やコード実装に入らないこと。
   - まずは必ず複数の解決策（案A, 案B, 案C...）を提示し、それぞれの Pros / Cons / 影響範囲 をフラットに比較する。
   - ユーザーが最初から1つの案に固執して深掘りしようとした場合、「詳細を詰める前に、他の選択肢も軽く洗い出しておきませんか？」と提案し、視野を広げるサポートをする。
2. **意思決定フェーズ:**
   - 複数の選択肢と Pros / Cons が出揃った段階で、「この状態でドキュメント（Notion等）にまとめ、チームに相談（壁打ち）しますか？」とユーザーに確認する。
3. **深さ優先探索 (DFS) フェーズ:**
   - チーム（またはユーザー自身）によって方針が1つに決定された後にのみ、詳細な設計、エッジケースの考慮、シーケンスの整理、具体的な実装コードの出力に移行する。

## ツール

### Git

- コミットメッセージは以下のルールに従う:
  - 1行目（件名）: 日本語で「〜する」「〜を修正する」のように動詞終わりで変更内容を簡潔に書く
    - prefix は基本的に不要。ただし revert/refactor など文脈上必要な場合は `revert:` `refactor:` のように小文字で付けてよい
  - 3行目以降（本文）: 必要に応じて「なぜその変更をしたのか」を書く
    - 変更の動機・背景・理由を説明する（「何を変えたか」ではなく「なぜ変えたか」）
    - 関連する GitHub PR のレビューコメント URL、Slack スレッド URL、Notion ページ URL などがあればそのまま貼る
    - 自明な変更や件名だけで十分な場合は本文なしでよい
    - GitHub上で suggestion されたものを採用する場合、 Co-Authored-By を利用してレビュワーをクレジットすること

### Pull Request

- Pull Request は必ず **ドラフト** (`gh pr create --draft`) で作成する
- レビュアーのアサインは自動で行わない（ユーザーが明示的に指示した場合のみ）

### コマンド

- GitHub に関する操作は gh コマンドを使用する
  - `github.com` のURLを渡されたときは、 gh コマンドを利用して情報を取得してください
  - Pull Request の作成を依頼されたときは、必ずリポジトリの Pull Request テンプレートファイルを探し、あれば利用してください
- Jira の情報取得は acli コマンドを使用する
  - `*.atlassian.net` の URL を渡されたとき、または Jira チケット ID（`ABC-123` のような `英大文字-数字` 形式）を渡されたときは、`/read-jira-workitem` スキルを呼び出し、acli コマンドで情報を取得すること
  - チケットの description を更新する場合は `/update-jira-workitem` スキルを使うこと

- Figma

ユーザーは JetBrains IDE を使用しており、 rails server や sidekiq あるいは pnpm dev といった開発用のサーバーはすでに起動済である。そのため、 コーディングエージェント自身がサーバーを起動する必要は無い。

### 外部サービス（Notion・Jira 等）の更新ルール

- Notion ページや Jira チケットなどの外部コンテンツを**更新・編集する前に、必ず最新の内容を再取得**すること
- 会話中にすでに取得済みの内容があっても、ユーザーが手動で変更している可能性があるため、更新直前に改めて読み込む
- これにより、ユーザーの手動変更を上書きしてしまう事故を防ぐ
